# Требования к визуализатору конфигурации линий

## 1. Формат и запуск
- Приложение должно запускаться локально без сборки.
- Базовый вариант: один файл `index.html` (включая `HTML + CSS + JS`).
- Допустимый вариант: несколько файлов (`.html/.css/.js`) без bundler/сборки, открытие напрямую в современном Chrome.

## 2. Макет интерфейса
- Двухпанельный интерфейс:
- Левая панель (основная, большая): зона визуализации конфигурации.
- Правая панель (управление): элементы ввода и переключатели режимов.

## 3. Ввод данных
- В правой панели должна быть `textarea` для вставки лога программы.
- Парсинг приоритетно из блока:
- `#LINES_BEGIN ... #LINES_END`.
- Внутри блока строки формата `m,b` (CSV), возможен заголовок `m,b`.
- Если блок не найден, включается fallback-парсинг:
- поиск строк с коэффициентами линий, в том числе формата `m,b`;
- (по возможности) поддержка строк формата `y = m * x + b`.

## 4. Геометрическая модель
- Каждая строка интерпретируется как прямая `y = m*x + b`.
- Нужно вычислить попарные точки пересечения (если прямые не параллельны).
- Область визуализации должна автоматически подбираться так, чтобы все найденные точки пересечения попадали в видимую часть.
- Должна быть обработка вырожденных случаев:
- параллельные/почти параллельные прямые;
- отсутствие пересечений.

## 5. Режимы отрисовки
- Простейший режим: рисуются линии.
- Флажок `Шахматная раскраска` (по умолчанию включен):
- вместо линий рисуются области (многоугольники), на которые плоскость разбивается набором линий;
- применяется двуцветная раскраска (черно/белая), допустимая как для planar-разбиения.
- Из двух возможных шахматных раскрасок выбирается та, в которой черных областей по количеству больше.
- Белые области можно не рисовать, так как фон должен быть белым.
- Флажок `Внешние области` (по умолчанию включен):
- если выключен, внешние черные области не рисовать;
- если включен, внешние черные области рисовать.

## 6. Навигация и трансформации
- Поддержать интерактив:
- перемещение (pan), желательно мышью (drag);
- масштабирование (zoom) колесом мыши;
- поворот (rotation).
- Геометрические трансформации задаются не сырыми полями (`rotation/scale/skew/...`), а командами-модификаторами:
- `Rotate ±10°`,
- `X ±10%`, `Y ±10%`,
- `Skew X ±6°`, `Skew Y ±6°`,
- `Mirror X`, `Mirror Y`.
- Команды интерпретируются в экранных координатах и применяются относительно центра viewport.
- Требование к композиции трансформаций:
- поворот не должен неявно вносить дополнительное сжатие/сдвиг;
- порядок применения трансформаций должен быть фиксирован и предсказуем.
- Нужна кнопка автоматического подбора zoom и pan (`Fit`):
- `Fit` подбирает максимальный zoom и центрирует камеру так, чтобы после текущей геометрической трансформации минимальный отступ от точек пересечения до границы viewport был 6%.
- Если точек пересечения нет, используется fallback по `bbox`.
- `Reset View` сбрасывает только камеру (`pan=0, zoom=1`) без изменения геометрии.
- `Reset Transform` сбрасывает только геометрическую матрицу.

## 7. Клиппинг и границы сцены
- Для внутренних и внешних областей нужен корректный геометрический клиппинг по viewport.
- Клиппинг обязателен для корректного экспорта и визуального соответствия экрану.
- Разбиение плоскости строится на большой рабочей области `sceneBasePoly`; эта область должна автоматически расширяться при приближении viewport к ее границе, чтобы пользователь визуально не упирался в искусственный прямоугольник.

## 8. Экспорт
- Нужен экспорт в SVG.
- В экспорт должен попадать ровно тот вид, который на экране после всех трансформаций.
- Для этого экспорт использует ту же геометрию/клиппинг, что и рендер.
- Перспектива: возможен будущий экспорт в EPS (сейчас можно не реализовывать, но архитектурно не блокировать).

## 9. Ограничения платформы
- Целевой браузер: современный Google Chrome.
- Совместимость со старыми браузерами не требуется.

## 10. Зафиксированная модель трансформаций
- Вводим два независимых слоя:
- `Geometry Transform` (трансформирует саму конфигурацию).
- `Camera/View` (панорамирование и zoom области просмотра).
- Это разделение нужно, чтобы поворот/скос/масштаб не смешивались с навигацией.

### 10.1 Параметры Geometry Transform
- Источник истины: матрица `M_geo` (2D affine, 2x3).
- UI задает команды-модификаторы, а не отдельные абсолютные поля геометрии.

### 10.2 Параметры Camera/View
- `viewPanX`, `viewPanY` (drag мышью).
- `viewZoom` (wheel).
- Кнопки:
- `Fit` (автоподбор zoom/pan по точкам пересечения, отступ 6%),
- `Reset View`,
- `Reset Transform`.

### 10.3 Порядок применения (фиксированный и предсказуемый)
- Экранная матрица:
- `M_screen = M_view * M_geo`.
- Для экранной команды `D_screen` геометрия обновляется как:
- `D_geo = M_view^{-1} * D_screen * M_view`,
- `M_geo := D_geo * M_geo`.
- Все рендер/клиппинг/экспорт используют один и тот же итог `M_screen`.

### 10.4 Ограничения и UX-заметки
- Трансформации должны оставаться обратимыми (не допускать вырождения матрицы).
- Для отладки полезно отображать текущую `M_geo` (read-only).

## 11. Статус и статистика областей
- В статусе выводить базовые счетчики (`lines`, `intersections`, `cells`, `external cells`, `zoom/pan`, `geo`).
- Дополнительно выводить распределение по числу сторон:
- черные: `b3, b4, ...`,
- белые: `w3, w4, ...`.
- Для внешних областей считать количество сторон как `finiteSides + 1`, где `finiteSides` — число граничных отрезков, лежащих на входных прямых.
- Классификацию `external/internal` определять при построении структуры ячеек (по системе полуплоскостей/знаков), а не по факту клиппинга рабочей областью.

## 12. Примеры формата входных данных

### 12.1 Предпочтительный формат (блок `#LINES`)
```text
#LOG_BEGIN
  620000 V= 1.537151e-04 G=  1.394e-02 step=  1.000e+00 wr=  6.899e-04 vS=  1 S=0
#LOG_END
#RESULT_BEGIN
SAT_FOUND strict=1 margin=0 worst_raw=-0.000637767 worst_shifted=0.00936223
SUMMARY n=23 gens=253 iter=650000 max_iter=inf V=0.000117894 GRAD=0.00392999 STEP=0.0498188 STRICT=1 MARGIN=0 worst_raw=-0.000637767 worst_shifted=0.00936223 viol_strict=0 viol_margin=105 best_V=0.000117894
#RESULT_END
#LINES_BEGIN
m,b
-28.3,-15.8
-13.6,2.18
-10.3,-18.7
-10.1,5.76
-8.77,3.62
#LINES_END
```

### 12.2 Fallback: просто список `m,b` без блоков
```text
-28.3,-15.8
-13.6,2.18
-10.3,-18.7
-10.1,5.76
-8.77,3.62
```

### 12.3 Fallback: строки вида `y = m * x + b`
```text
y = -28.3 * x -15.8
y = -13.6 * x  +2.18
y = -10.3 * x -18.7
y = -10.1 * x  +5.76
y = -8.77 * x  +3.62
```

### 12.4 Замечания по парсингу
- Игнорировать пустые строки и посторонний текст.
- Для CSV-формы поддерживать optional заголовок `m,b`.
- Для формы `y = ...` поддерживать пробелы и знак перед `b`.
