# Лог оптимизаций p3_maximal_v11.c

Дата: 2026-03-23

## Методика измерений

Два бенчмарка, замеряются после каждого изменения:

| Бенчмарк | Команда | Что измеряет |
|-----------|---------|-------------|
| n=18 full | `./p3v11 18` | Полный перебор, 255 результатов, 57393 нод |
| n=34 first | `./p3v11 34 1` | Первый результат, 336638 нод |

Метрика: **μs/нод** = время / ноды. Ноды не должны меняться (оптимизации per-node, не алгоритмические).

Компиляция: `gcc -O2 -o p3v11 p3_maximal_v11.c`

Платформа: WSL2 Linux 6.6.87, gcc, x86_64.

## Baseline (до оптимизаций, после перехода N_MAX 32→64 и uint32→uint64)

| n=18 | n=34 |
|------|------|
| 2.14s, 57393 нод, **37.2 μs/нод** | 18.4s, 336638 нод, **54.6 μs/нод** |

## 1. UP+MRV merge — объединение unit propagation с MRV выбором ✓

**Гипотеза**: В оригинале после unit propagation делается второй полный скан всех гиперлиний для выбора branching hyperline (MRV). Это дублирование — последний раунд UP уже сканирует все гиперлинии и может собрать MRV-информацию одновременно.

**Реализация**: Трекинг best_p/best_count/best_cands прямо в цикле UP. Секция "Find branching hyperline" удалена.

**Результат**:

| n=18 | n=34 |
|------|------|
| 1.95s, **34.0 μs/нод** | 15.7s, **46.7 μs/нод** |

Ускорение: **-9% (n=18), -14% (n=34)**. На большем n эффект сильнее (больше гиперлиний в скане).

## 2. inline chi_put ✓

**Гипотеза**: chi_put вызывается ~n раз на каждое размещение. Без inline — overhead вызова функции (save/restore регистров). Добавление `static inline` позволяет компилятору встроить код.

**Реализация**: `static int chi_put(...)` → `static inline int chi_put(...)`

**Результат** (поверх #1):

| n=18 | n=34 |
|------|------|
| 1.81s, **31.5 μs/нод** | 14.7s, **43.7 μs/нод** |

Ускорение: **дополнительно -14% (n=18), -7% (n=34)**. Суммарно с #1: **-15% (n=18), -20% (n=34)**.

## 3. -O3 вместо -O2 ✗

**Гипотеза**: Более агрессивные оптимизации компилятора.

**Результат**: n=18: 1.81s (без изменений), n=34: 14.6s (marginally better). Не стабильно лучше — -O3 раздувает код, хуже для icache. **Отброшено.**

## 4. Canonical chi_v (1 запись вместо 6) ✗

**Гипотеза**: chi_put пишет 6 перестановок `chi_v[i][j][k]` при установке тройки. Каноническая форма (хранить только i≤j≤k, вычислять знак) сократит writes с 6 до 1, chi_undo с 6 до 1 стирания. Экономия на cache pressure.

**Реализация**: Функция `chi_canon(i,j,k)` → нормализованные (a,b,c) + sign. chi_put пишет одну ячейку. chi_undo стирает одну ячейку. Чтение через `chi_get()`.

**Результат**: n=18: 1.86s (**хуже** на 3%), n=34: 14.4s (без изменений).

**Причина**: Для текущих n массив chi_v (274KB) помещается в L2 cache. 6 прямых записей дешевле, чем 6 сравнений + sign computation при каждом chi_put. Overhead нормализации ≥ экономия на записях. Может стать актуально для n>50. **Отброшено.**

## 5. Dirty-set / worklist-based unit propagation ✗

**Гипотеза**: После forced placement в UP достаточно пересканировать только "грязные" гиперлинии (те, где pred_mask/succ_mask или adj изменились), а не все n-1.

### 5a. Post-hoc dirty из trail ✗ (частично работает)

**Реализация**: После forced placement, вычислять dirty_mask из ord_trail и adj_trail. Dirty scan → если ничего не нашёл → full scan для MRV + safety.

**Результат**: n=18: 1.58s (56789 нод), n=34: 10.5s (326903 нод).

По μs/нод: ускорение. Но **ноды изменились** (57393→56789, 336638→326903) — dirty scan пропускает cross-hyperline adj эффекты, находит forced moves в другом порядке, дерево меняется. Результаты верны (255, 1), но не детерминистично. **Отброшено как костыльное.**

### 5b. Worklist через callbacks в update_ordering/add_adj ✗

**Реализация**: up_worklist (uint64_t bitmask) обновляется прямо в update_ordering (при новом ordering constraint) и add_adj (при изменении nbr_count). Для cross-hyperline adj: при nbr_count→2, сканируются hyperlines где элемент является frontier.

**Результат**: n=10: 0 результатов (баг), n=18: 0 результатов. Причина: static cached candidate arrays перезаписываются рекурсивными вызовами solve().

Упрощённая версия (worklist для UP, full scan для MRV): n=18: 156631 нод (должно быть 57393). **Ноды в 3× больше** — worklist-уведомления неполны, пропускаются forced moves, дерево раздувается.

**Причина фундаментальная**: cross-hyperline adj зависимости — can_add_adj(e, p, f) проверяет nbr_count[e][...], где e — кандидат на гиперлинии p. Изменение nbr_count на гиперлинии q может повлиять на кандидатов ЛЮБОЙ гиперлинии, где q является remaining элементом. Это фактически все гиперлинии, что убивает идею dirty-set.

**Вывод**: Корректный worklist требует уведомления всех гиперлиний при каждом adj изменении, что эквивалентно полному скану. **Отброшено.**

## 6. Block grouping ~0%

**Гипотеза**: Когда два элемента на гиперлинии известно-соседи и их порядок определён, они образуют "блок". Только эндпоинты блоков — валидные кандидаты. Замена is_block_endpoint (O(neighbors)) на O(1) проверку через blk_prev/blk_next.

**Реализация**:
- `blk_next[p][e]`, `blk_prev[p][e]` — цепочки внутри блоков
- `blk_other_end[p][e]` — для эндпоинтов, указывает на другой конец блока
- `try_blk_merge(q, a, b)` — вызывается из add_adj и update_ordering при обнаружении adj+ordering
- Trail-based undo через BlkTrailEntry (p, a, b, ll, rr)
- В get_left_candidates: `blk_prev[p][e] != -1 && in rem` → skip
- В get_right_candidates: `blk_next[p][e] != -1 && in rem` → skip
- Не мутируем блоки при placement — проверяем rem_mask (Option B)

**Результат**:

| n=18 | n=34 |
|------|------|
| 1.82s, 57393 нод, **31.7 μs/нод** | 14.8s, 336638 нод, **43.8 μs/нод** |

Ноды идентичны — корректно. Но ускорения нет: старая is_block_endpoint (проверка 0-2 соседей) уже была O(1) в среднем, а try_blk_merge добавляет overhead на каждый add_adj/update_ordering. Баланс ~0.

Потенциально полезно на больших n (>40), где блоки длиннее и фильтруют больше не-эндпоинтов из rem_mask итерации.

## 7. Early contradiction detection из блоков ✗

**Гипотеза**: После merge блока проверить `!(blk_left_ep[q] & rq) || !(blk_right_ep[q] & rq)` — если на гиперлинии не осталось left или right эндпоинтов среди remaining → немедленный contradiction.

**Результат**: n=10: 0 результатов (должно 3), n=18: 0 результатов. **Ложные срабатывания.**

**Причина**: Проверка `blk_left_ep[q] & rq` не учитывает, что после placement элемента его сосед по блоку СТАНОВИТСЯ новым эндпоинтом (динамически, через rem_mask). В момент merge проверка статична и видит "нет эндпоинтов", хотя после размещения промежуточных элементов эндпоинты появятся. **Отброшено.**

## 8. Bitmask эндпоинтов ✗

**Гипотеза**: Поддерживать `blk_left_ep[p]` / `blk_right_ep[p]` (uint64_t). В get_left_candidates итерировать по `blk_left_ep[p] & rem_mask[p]` вместо `rem_mask[p]` — пропускать внутренние элементы блоков.

**Результат**: n=10: 0 результатов, n=18: 0 результатов (9.5M нод вместо 57K).

**Причина**: Когда элемент e размещается (удаляется из rem_mask), его сосед b по блоку должен стать новым эндпоинтом. Динамическая проверка `blk_prev[p][e] == -1 || !(rp & prev)` обрабатывала это автоматически через rem_mask. Bitmask blk_left_ep не обновляется при placement → элемент b остаётся отфильтрованным. Для корректности нужно обновлять blk_left_ep/blk_right_ep в place_left/place_right + trail для undo — слишком сложно для скромного выигрыша. **Отброшено.**

## Нереализованные идеи

- **PGO (Profile-Guided Optimization)**: двухпроходная компиляция gcc -fprofile-generate / -fprofile-use. Типично 10-20% на branchy коде. Не требует изменений кода.
