# Алгоритм построения p3-максимальных конфигураций псевдопрямых

Реализация: `p3_maximal_v11.c`
Источник: Bokowski, Roudneff, Strempel — "Cell Decompositions of the Projective Plane", Section 4

## Задача

Есть n прямых на проективной плоскости (можно думать: прямые на обычной плоскости + "прямая на бесконечности"). Каждая пара прямых пересекается ровно в одной точке. Мы хотим расположить их так, чтобы количество **треугольников** (областей, ограниченных ровно тремя прямыми) было максимальным: ровно n(n-1)/3.

## Что такое гиперлиния

Возьмём одну прямую, скажем прямую номер p. На ней лежат n-1 точек пересечения (по одной с каждой другой прямой). Эти точки идут в каком-то **круговом порядке** вдоль прямой (круговом — потому что проективная прямая замкнута, как окружность).

**Гиперлиния p** — это просто этот круговой порядок точек на прямой p. Например, для 6 прямых гиперлиния 1 может быть [6, 4, 5, 2, 3] — это значит, что двигаясь вдоль прямой 1, мы встречаем пересечения с прямыми 6, 4, 5, 2, 3, и потом снова 6 (круг).

Задать расположение всех n прямых — это задать все n гиперлиний (круговых порядков).

## Когда три прямые образуют треугольник

Тройка прямых (i, j, k) образует треугольник тогда и только тогда, когда:
- на прямой i точки пересечения с j и k **соседние** (стоят рядом в круговом порядке)
- на прямой j точки пересечения с i и k соседние
- на прямой k точки пересечения с i и j соседние

То есть все три пары должны быть "рядом" — каждая на своей прямой.

## Алгоритм (общая картина)

**Ключевая идея**: строим все гиперлинии **одновременно**.

1. **Начало.** Одну гиперлинию (прямую n — "на бесконечности") фиксируем полностью. Для каждой остальной гиперлинии p известны три элемента: первый (n), второй и последний.

   Откуда они известны: в гиперлинии n у элемента p есть два соседа — назовём их second[p] и last[p]. Соседство p и second[p] в гиперлинии n означает треугольник (n, p, second[p]). Условие треугольника требует, чтобы n и second[p] были соседями в гиперлинии p. Элемент n стоит первым, значит second[p] — второй. Аналогично из треугольника (n, p, last[p]) следует, что last[p] — сосед n с другой стороны, то есть последний элемент. Всё, что между вторым и последним — неизвестно, это и заполняет алгоритм.

2. **Основной цикл.** На каждом шаге:
   - Смотрим на все незавершённые гиперлинии
   - Выбираем **самую ограниченную** — ту, где меньше всего вариантов, какой элемент поставить следующим (MRV — Minimum Remaining Values)
   - Если вариант один — ставим без раздумий (**forced move**, unit propagation)
   - Если вариантов несколько — пробуем каждый (с откатом при неудаче)

3. **Магия распространения.** Когда мы ставим элемент e рядом с элементом f на прямой p, это значит, что прямые f и e **соседние на прямой p** — образуется треугольник (p, f, e). А треугольник — это ограничение сразу на **три** гиперлинии! Мы узнаём:
   - на прямой f: прямые p и e тоже должны быть соседними
   - на прямой e: прямые p и f тоже должны быть соседними

   Это резко сужает варианты для других гиперлиний. Одно решение на одной прямой каскадом ограничивает всё остальное.

4. **Проверка соседств.** У каждого элемента на каждой прямой ровно 2 соседа (круг). Если из-за треугольников у элемента набирается 3 "обязательных соседа" — противоречие, откатываемся.

## Что такое ноды

**Нода (node)** — это одна точка принятия решений в дереве перебора. Каждый раз, когда алгоритм выбирает, какой элемент поставить следующим, это одна нода. Меньше нод = быстрее работает.

Для n=16: наивный подход (гиперлинии по одной) даёт миллиарды нод. Алгоритм из статьи находит решение за **23 ноды** — потому что каскад ограничений от треугольников оставляет почти на каждом шаге единственный вариант.

## Двустороннее заполнение (v10+)

### Идея

В v9 гиперлинии заполнялись только слева направо:

```
прямая p: [n, second, ..., f, ?, ?, ..., last]
                            ^
                   приклеиваем сюда
```

В v10+ заполнение идёт **с двух сторон** — и слева, и справа, навстречу друг другу:

```
прямая p: [n, second, ..., f_L, ?, ?, ..., f_R, ..., last]
                            ^                ^
                    left frontier      right frontier
```

`left_arr[p]` хранит элементы слева (начиная с n, second, ...),
`right_arr[p]` хранит элементы справа (начиная с last, ...),
`rem_mask[p]` — битовая маска оставшихся элементов в "дырке" между фронтирами.

### Зачем это нужно

Элемент может быть кандидатом слева, но не справа (или наоборот). Если у него есть невыставленные предшественники (не может идти следующим слева) — он ещё может быть следующим справа. Двусторонность даёт:
- Больше forced moves (кандидат один слева — ставим, даже если справа их несколько)
- Более точный MRV (считаем кандидатов с обеих сторон)
- Раннее обнаружение тупиков (0 кандидатов с обеих сторон = противоречие)

### Каноническая форма

Чтобы избежать дублирования (одна и та же конфигурация может быть найдена через разные порядки left/right заполнения), при ветвлении (branching) используются **только левые кандидаты**. Правые кандидаты используются только для forced moves (когда их ровно 1) или когда левых кандидатов 0.

### Chi-знаки при размещении

**place_left(p, e)** — e ставится после left frontier:
- Chi: e **после** всех left elements → `chi(p, x, e) = sgn[p][x] * sgn[p][e]`
- Chi: e **до** всех right elements и remaining → `chi(p, x, e) = -(sgn[p][x] * sgn[p][e])`

**place_right(p, e)** — e ставится перед right frontier:
- Chi: e **после** всех left elements и remaining → `chi(p, x, e) = sgn[p][x] * sgn[p][e]`
- Chi: e **до** всех right elements → `chi(p, x, e) = -(sgn[p][x] * sgn[p][e])`

Когда дырка закрывается (`rem_mask == 0`), left frontier и right frontier оказываются соседями — регистрируется дополнительный треугольник.

## Предвычисленные множества предшественников/последователей (v11)

### Проблема в v10

Для проверки, может ли элемент e быть следующим кандидатом слева на гиперлинии p, нужно убедиться, что нет "невыставленных предшественников" — элементов, которые должны стоять раньше e. В v10 это делалось перебором **всех** оставшихся элементов:

```c
for x in rem[p]:
    if chi_v[p][x][e] != 0 && chi_v[p][x][e] == sgn[p][x] * sgn[p][e]:
        // x должен стоять до e → e не кандидат
```

Это O(|rem|) на кандидата, O(|rem|²) на гиперлинию.

### Решение в v11: битовые маски

Для каждого элемента e на гиперлинии p хранятся:
- `pred_mask[p][e]` — битовая маска элементов, которые должны стоять **до** e
- `succ_mask[p][e]` — битовая маска элементов, которые должны стоять **после** e

Проверка кандидата — одна битовая операция:
```c
if (pred_mask[p][e] & rem_mask[p])  // есть невыставленные предшественники?
    continue;                        // e не кандидат слева
```

Аналогично для правых кандидатов:
```c
if (succ_mask[p][e] & rem_mask[p])  // есть невыставленные последователи?
    continue;                        // e не кандидат справа
```

### Как обновляются маски

При каждом `chi_put(i, j, k, val)`, который записывает новое значение, вызывается `update_ordering()` для **трёх** гиперлиний:
- Гиперлиния i: определён порядок j и k
- Гиперлиния j: определён порядок i и k
- Гиперлиния k: определён порядок i и j

Для каждой гиперлинии h с элементами a и b:
- Если оба a и b в `rem_mask[h]`:
  - Если a до b: `pred_mask[h][b] |= (1 << a)`, `succ_mask[h][a] |= (1 << b)`
  - Если b до a: `pred_mask[h][a] |= (1 << b)`, `succ_mask[h][b] |= (1 << a)`

### Автоматическая работа с rem_mask

Ключевое преимущество битовых масок: при удалении элемента из rem (placement) не нужно обновлять pred/succ маски — `& rem_mask` автоматически фильтрует уже размещённые элементы.

### Откат

Изменения масок записываются в `ord_trail` (тройки h, a, b). При undo:
```c
pred_mask[h][b] &= ~(1u << a);
succ_mask[h][a] &= ~(1u << b);
```

## Фильтр блочных конечных точек (v10+)

Когда из треугольников становится известно, что элементы a и b соседи на гиперлинии p (оба ещё в rem), они образуют **цепочку** (блок). Элемент внутри цепочки (у которого 2 известных соседа среди remaining) не может быть приклеен ни к левому, ни к правому фронтиру — только крайние элементы цепочки являются кандидатами.

```c
is_block_endpoint(p, e):
    // Считаем, сколько соседей e среди remaining
    // Если >= 2 → e внутри цепочки → не кандидат
```

## Хранение данных

### Chi (хиротоп)

Трёхмерный массив `int8_t chi_v[N+1][N+1][N+1]`. Значения: +1, -1, или 0 (не знаем). Доступ O(1).

При записи chi(a, b, c) = val пишем во все 6 перестановок:
```
chi_v[a][b][c] = +val,   chi_v[a][c][b] = -val
chi_v[b][a][c] = -val,   chi_v[b][c][a] = +val
chi_v[c][a][b] = +val,   chi_v[c][b][a] = -val
```

### Соседства (adjacency)

`nbr_count[q][x]` (0, 1 или 2) и `nbr_data[q][x][2]` — известные соседи элемента x на прямой q.

`can_add_adj(q, a, b)` — проверка: можно ли сделать a и b соседями (не превысят ли они лимит 2 соседей).

`add_adj(q, a, b)` — запись соседства + trail для undo.

### Remaining set

`uint32_t rem_mask[p]` — битовая маска оставшихся элементов. Итерация по установленным битам:
```c
while (mask) {
    int e = __builtin_ctz(mask);  // младший установленный бит
    mask &= mask - 1;             // сбросить его
    ...
}
```

## Откат: trail

Все три структуры (chi, adj, ordering) используют **trail** — список сделанных изменений.

- **chi_trail**: тройки (i, j, k), при undo обнуляем все 6 перестановок
- **adj_trail**: пары (q, x), при undo декрементим `nbr_count[q][x]`
- **ord_trail**: тройки (h, a, b), при undo очищаем биты в pred/succ масках

Операции:
- `mark()` — вернуть текущую позицию trail (точка сохранения)
- `undo(mark)` — откатить все изменения до сохранённой позиции

```c
int cmk = chi_mark(), amk = adj_mark(), omk = ord_mark();
if (place(p, e)) {
    solve();                    // рекурсия
}
unplace(p, e);
chi_undo(cmk);
adj_undo(amk);
ord_undo(omk);
```

## Основной цикл (solve)

```
solve():
  // 1. Unit propagation: делать forced moves пока есть
  while true:
    for p = 1 to n-1:
      if rem[p] пуст — skip
      lcands = get_left_candidates(p)
      rcands = get_right_candidates(p)

      if lcands + rcands == 0 → contradiction, break

      // Forced move: один кандидат на какой-либо стороне
      if lcands == 1 → place_left(p, lcands[0])
      else if rcands == 1 → place_right(p, rcands[0])
      else if lcands + rcands == 1 → place единственный

    if не нашли forced move → break

  // 2. Все гиперлинии завершены? → решение найдено
  if нет незавершённых → проверить tri count, сохранить

  // 3. Ветвление: выбрать гиперлинию с min кандидатов (MRV)
  //    Каноническая форма: ветвимся по LEFT кандидатам
  //    (right — только если left пуст)
  for each candidate e:
    mark trails
    place(p, e)
    solve()                    // рекурсия
    unplace(p, e)
    undo trails
```

## Проверка кандидатов

### get_left_candidates(p)

Для каждого e в rem[p]:
1. **Block-endpoint filter**: пропустить e, если у него ≥ 2 соседей среди remaining (внутренний элемент цепочки)
2. **Adjacency filter**: проверить `can_add_adj(p, f, e)`, `can_add_adj(f, p, e)`, `can_add_adj(e, p, f)` — треугольник (p, f, e) не приведёт к >2 соседей?
3. **Predecessor check (v11)**: `(pred_mask[p][e] & rem_mask[p]) == 0` — нет невыставленных предшественников?

### get_right_candidates(p)

Аналогично, но:
- Фронтир — `right_arr[p][right_len[p]-1]`
- Successor check: `(succ_mask[p][e] & rem_mask[p]) == 0`

## Размещение элемента

### place_left(p, e)

1. Убрать e из rem_mask[p]
2. Добавить e в left_arr[p]
3. Установить chi-значения:
   - e после всех left elements
   - e до всех right elements и remaining
4. Зарегистрировать треугольник (p, f, e) через apply_triangle
5. Если дырка закрылась — треугольник (p, e, right_frontier)

### apply_triangle(p, a, b)

Треугольник (p, a, b) означает три пары соседств:
- a и b — соседи на прямой p
- p и b — соседи на прямой a
- p и a — соседи на прямой b

## Симметрия

Гиперлиния n имеет циклическую симметрию: поворот на 2 позиции эквивалентен перенумерации σ(i) = i+1 (mod n-1). Группа Z_{n-1} действует на конфигурации. Для n=22 (n-1=21) каждая уникальная конфигурация встречается до 21 раза (или меньше, если у неё есть нетривиальный стабилизатор). Symmetry breaking пока не реализован.
