# Алгоритм построения p3-максимальных конфигураций псевдопрямых

Реализация: `p3_maximal_v9.py`
Источник: Bokowski, Roudneff, Strempel — "Cell Decompositions of the Projective Plane", Section 4

## Задача

Есть n прямых на проективной плоскости (можно думать: прямые на обычной плоскости + "прямая на бесконечности"). Каждая пара прямых пересекается ровно в одной точке. Мы хотим расположить их так, чтобы количество **треугольников** (областей, ограниченных ровно тремя прямыми) было максимальным: ровно n(n-1)/3.

## Что такое гиперлиния

Возьмём одну прямую, скажем прямую номер p. На ней лежат n-1 точек пересечения (по одной с каждой другой прямой). Эти точки идут в каком-то **круговом порядке** вдоль прямой (круговом — потому что проективная прямая замкнута, как окружность).

**Гиперлиния p** — это просто этот круговой порядок точек на прямой p. Например, для 6 прямых гиперлиния 1 может быть [6, 4, 5, 2, 3] — это значит, что двигаясь вдоль прямой 1, мы встречаем пересечения с прямыми 6, 4, 5, 2, 3, и потом снова 6 (круг).

Задать расположение всех n прямых — это задать все n гиперлиний (круговых порядков).

## Когда три прямые образуют треугольник

Тройка прямых (i, j, k) образует треугольник тогда и только тогда, когда:
- на прямой i точки пересечения с j и k **соседние** (стоят рядом в круговом порядке)
- на прямой j точки пересечения с i и k соседние
- на прямой k точки пересечения с i и j соседние

То есть все три пары должны быть "рядом" — каждая на своей прямой.

## Наивный подход (v7) — медленный

Строим гиперлинии **по одной**: сначала полностью определяем порядок на прямой 1, потом на прямой 2, и т.д. Проверяем совместимость на ходу. Проблема: когда строим прямую 1, мы ничего не знаем про прямые 2–15, и перебор слишком большой.

## Алгоритм из статьи (v9) — быстрый

**Ключевая идея**: строим все гиперлинии **одновременно**.

1. **Начало.** Одну гиперлинию (прямую n — "на бесконечности") фиксируем полностью. Для каждой остальной гиперлинии p известны три элемента: первый (n), второй и последний.

   Откуда они известны: в гиперлинии n у элемента p есть два соседа — назовём их second[p] и last[p]. Соседство p и second[p] в гиперлинии n означает треугольник (n, p, second[p]). Условие треугольника требует, чтобы n и second[p] были соседями в гиперлинии p. Элемент n стоит первым, значит second[p] — второй. Аналогично из треугольника (n, p, last[p]) следует, что last[p] — сосед n с другой стороны, то есть последний элемент. Всё, что между вторым и последним — неизвестно, это и заполняет алгоритм.

2. **Основной цикл.** На каждом шаге:
   - Смотрим на все незавершённые гиперлинии
   - Выбираем **самую ограниченную** — ту, где меньше всего вариантов, какой элемент поставить следующим
   - Если вариант один — ставим без раздумий (**forced move**)
   - Если вариантов несколько — пробуем каждый (с откатом при неудаче)

3. **Магия распространения.** Когда мы ставим элемент e рядом с элементом f на прямой p, это значит, что прямые f и e **соседние на прямой p** — образуется треугольник (p, f, e). А треугольник — это ограничение сразу на **три** гиперлинии! Мы узнаём:
   - на прямой f: прямые p и e тоже должны быть соседними
   - на прямой e: прямые p и f тоже должны быть соседними

   Это резко сужает варианты для других гиперлиний. Одно решение на одной прямой каскадом ограничивает всё остальное.

4. **Проверка соседств.** У каждого элемента на каждой прямой ровно 2 соседа (круг). Если из-за треугольников у элемента набирается 3 "обязательных соседа" — противоречие, откатываемся.

## Что такое ноды

**Нода (node)** — это одна точка принятия решений в дереве перебора. Каждый раз, когда алгоритм выбирает, какой элемент поставить следующим, это одна нода. Меньше нод = быстрее работает.

Для n=16: наивный подход не справляется (миллиарды нод). Алгоритм из статьи находит решение за **132 ноды** — потому что каскад ограничений от треугольников оставляет почти на каждом шаге единственный вариант.

## Как устроен перебор подробнее

### Что мы реально храним

Для каждой прямой p мы строим её круговой порядок **слева направо**:

```
прямая p: [n, second, ?, ?, ?, ..., ?, last]
                       <-- заполняем -->
```

Известны: первый элемент (n), второй (second), последний (last). Между second и last — дырка, которую мы заполняем **строго слева направо**, приклеивая по одному элементу к правому краю заполненной части.

### Что даёт нам один шаг

Когда мы приклеиваем e к правому краю прямой p (рядом с f):

```
прямая p: [n, second, ..., f, e, ?, ..., last]
                              ^ новое соседство
```

Мы узнаём **не позицию** p на прямой e, а **относительный порядок**: кто идёт раньше, а кто позже на каждой другой прямой.

### Как работает "знание о порядке"

Для каждой тройки прямых (a, b, c) есть значение **chi(a, b, c) = +/-1**, которое отвечает на вопрос:

> "На прямой a, точка пересечения с b стоит до или после точки пересечения с c?"

Это как ориентация треугольника: по часовой или против.

Когда мы ставим e после f на прямой p, мы **определяем** chi(p, x, e) для всех x, которые уже стоят на прямой p — потому что теперь мы знаем, что все они идут раньше e.

Но chi — **общий** для всех прямых. Значение chi(p, f, e) — это та же тройка элементов, что chi(f, p, e) и chi(e, p, f), просто с другим знаком. Поэтому, определив порядок на прямой p, мы автоматически узнаём что-то про порядок на прямых f и e.

### Конкретный пример

Прямые 1–6, строим прямую 1. Приклеиваем 5 после 4:

```
прямая 1: [6, 4, 5, ?, 3]
```

Мы узнали: на прямой 1 элемент 4 стоит до 5. Это задаёт chi(1, 4, 5).

Теперь chi(5, 1, 4) тоже определён (через ту же тройку). Это значит: **на прямой 5** мы теперь знаем взаимный порядок элементов 1 и 4. А на прямой 5 уже стоят какие-то элементы, и среди оставшихся кандидатов — 1 и 4. Если из нового ограничения следует, что 4 должен идти раньше 1 на прямой 5, то 1 уже **не может** быть следующим кандидатом на прямой 5 (пока 4 не поставлен).

### Как это сужает перебор

На прямой e мы **не решаем**, куда поставить p. Мы просто копим знания:

- "на прямой e элемент 1 идёт раньше элемента 4" — из одного шага
- "на прямой e элемент 3 идёт раньше элемента 1" — из другого шага
- ...

Когда наступает очередь прямой e, эти накопленные ограничения определяют: какой элемент **может** быть следующим (у него нет "обязательных предшественников" среди оставшихся). Часто вариант один — и шаг делается без перебора.

### Плюс ограничение на соседства

Поверх всего: если мы знаем, что треугольник (p, f, e) существует, то прямые p и f **соседние** на прямой e. Но у каждого элемента на прямой ровно 2 соседа. Если у элемента p на прямой e уже набралось 2 соседа, а треугольник требует третьего — противоречие, откат. Это самое мощное ограничение: для n=10 оно сократило перебор в 260 раз.

## Хранение данных и откат

### Хранение chi

Chi хранится в трёхмерном массиве `v[a][b][c]` размера (n+1)^3. Значения: +1, -1, или 0 (не знаем). Доступ — за одно обращение к массиву, без вычислений.

При записи chi(a, b, c) = +1 мы сразу пишем во все 6 перестановок:

```
v[a][b][c] = +1
v[a][c][b] = -1    <- поменяли два аргумента — знак меняется
v[b][a][c] = -1
v[b][c][a] = +1
v[c][a][b] = +1
v[c][b][a] = -1
```

Так при чтении не надо ничего сортировать — просто `v[a][b][c]`.

### Хранение соседств

Для каждого элемента x на каждой прямой q храним список его известных соседей (максимум 2). Реализовано как массив `nbr_data[q][x]` длины 2 и счётчик `nbr_count[q][x]`.

### Как откатываем: trail

Обе структуры используют **trail** (след) — просто список сделанных изменений.

Для chi:

```python
trail = []

def put(a, b, c, val):
    if v[a][b][c] != 0:          # уже записано
        return v[a][b][c] == val  # совпадает? ОК. Нет? Противоречие.
    v[a][b][c] = val              # пишем во все 6 перестановок
    v[a][c][b] = -val
    ...
    trail.append((a, b, c))       # запоминаем, что записали
    return True

def mark():
    return len(trail)             # "сохранить позицию"

def undo(saved_mark):
    while len(trail) > saved_mark:
        a, b, c = trail.pop()    # берём последнюю запись
        v[a][b][c] = 0           # обнуляем все 6 перестановок
        v[a][c][b] = 0
        ...
```

Для соседств — то же самое, только trail хранит `(q, x)` и откат делает `nbr_count[q][x] -= 1`.

### Как это используется в переборе

```python
mark_chi = chi.mark()          # запомнили состояние
mark_adj = adj_mark()

if place(p, e):                # поставили элемент, записали chi и соседства
    solve()                    # рекурсия — пошли глубже

unplace(p, e)                  # убрали элемент из списка
chi.undo(mark_chi)             # всё, что записали после mark — стёрли
adj_undo(mark_adj)             # соседства тоже стёрли
```

Это как Ctrl+Z — мы точно знаем, сколько действий сделали после метки, и отменяем их в обратном порядке.

### Почему не snapshot

Альтернатива — копировать весь массив перед каждым шагом (`snap = copy(data)`). Так делал v7. Для маленьких задач это нормально, но для n=16 массив chi — 17^3 = 4913 ячеек, копировать его на каждой из тысяч нод дорого.

Trail записывает только **изменения** (обычно 15–30 значений за шаг), поэтому и mark, и undo работают за время, пропорциональное числу новых записей, а не размеру массива.
