# Оптимизации из статьи Bokowski-Roudneff-Strempel (Section 4)

Текущая реализация: `p3_maximal_v11.c`
Статья: `sources/PL00009299.pdf`, стр. 383–385

## 1. ✅ Предвычисленные множества предшественников (реализовано в v11)

**Что в статье**: Для каждого элемента j на гиперлинии p хранятся множества:
- `b_k^j` — элементы, которые должны стоять **до** j (left predecessors)
- `a_l^j` — элементы, которые должны стоять **после** j (right successors)

**Реализация в v11**: Вместо счётчиков используются битовые маски `pred_mask[p][e]` и `succ_mask[p][e]` (uint32_t). При каждом `chi_put(i,j,k,val)` функция `update_ordering()` обновляет маски для всех трёх затронутых гиперлиний. Проверка кандидата — одна битовая операция: `(pred_mask[p][e] & rem_mask[p]) == 0`. Trail-based undo для масок через `ord_trail`.

**Результат**: Ускорение 1.7–1.8x (n=18: 3s→1.66s, n=22/10: 10s→5.9s). Число нод идентично v10.

## 2. Группировка элементов в блоки (СРЕДНИЙ импакт)

**Что в статье**: Когда из треугольника на другой гиперлинии становится известно, что a и b соседи на гиперлинии p, они объединяются в **блок** — упорядоченную цепочку `{..., a, b, ...}`. Каждый блок имеет:
- `Δ_{p,j}^{ml}` — тип треугольника на левом краю блока
- `Δ_{p,j}^{mr}` — тип треугольника на правом краю блока
- `cl_p^j` — можно ли приклеить блок слева
- `cr_p^j` — можно ли приклеить блок справа

**Что сейчас**: `is_block_endpoint()` проверяет на лету, является ли элемент крайним в цепочке соседств среди remaining. Это корректно, но пересчитывается при каждом вызове `get_left_candidates`.

**Что даст**:
- Кандидаты считаются по блокам, а не по элементам → более точный MRV
- Быстрое обнаружение тупиков: блок, который нельзя приклеить ни с одного конца → contradiction
- Приклеивание блока целиком = меньше нод (хотя unit propagation компенсирует это)

**Как реализовать**:
1. Union-find `blk_root[p][e]` для объединения элементов в блоки
2. `blk_end1[p][root]`, `blk_end2[p][root]` — концы цепочки
3. При `add_adj(q, a, b)` где оба в rem[q]: вызвать `merge_blocks(q, a, b)`
4. Trail-based undo для блочных операций
5. В `get_*_candidates`: итерировать по блокам, пробовать оба конца

**Сложность реализации**: СРЕДНЯЯ. Основная трудность — корректный undo при откате merge.

**Ожидаемое ускорение**: 10-30% для n≥22 (меньше лишних проверок, точнее MRV).

## 3. Boolean-переменные ориентации треугольников (НИЗКИЙ импакт)

**Что в статье**: `Δ_p^l` и `Δ_p^r` — булевы переменные, указывающие ориентацию (upright/inverted) последнего треугольника на левом и правом краях гиперлинии p.

**Что сейчас**: Ориентация неявно закодирована в chi-значениях и sgn-массиве.

**Что даст**: Быстрая проверка совместимости при приклеивании. Вместо chi-lookup — сравнение одного бита. Но chi-lookup уже O(1) (прямой доступ к массиву), так что выигрыш минимален.

**Сложность реализации**: НИЗКАЯ.

**Ожидаемое ускорение**: <5%.

## Порядок реализации

1. **Предвычисленные predecessors** (#1) — максимальный эффект, но сложно
2. **Группировка блоков** (#2) — средний эффект, средняя сложность
3. **Δ-переменные** (#3) — не стоит усилий при текущей производительности
